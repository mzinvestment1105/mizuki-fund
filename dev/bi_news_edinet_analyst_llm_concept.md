# BI → アナリスト（LLM）連携コンセプト

PMメモ（方向性の記録。実装詳細は別途パイプライン・エージェント定義に落とす）。

## いまの前提

- **個別銘柄の ETL** は既存パイプラインで取れるようになってきている（株価・財務など定量系の土台）。
- 次のレイヤーは **テキスト・開示系** を同じ「BIが集めて正規化する」枠に載せること。

## 取り込みたいデータソース（優先）

| ソース | 内容 | 役割 |
|--------|------|------|
| **立花証券 API** | ニュース | イベント・市場ストーリーのインプット |
| **EDINET** | **有価証券報告書** 等の開示 | ファンダ・リスク・事業内容の一次情報（開発中パイプラインを本番運用レベルへ） |

## 目標アーキテクチャ（概念）

```
[ 立花 ニュース ] ──┐
[ EDINET 開示   ] ──┼──► BI（ETL・格納・カタログ）──► 構造化/参照可能なデータ
[ 既存 個別ETL  ] ──┘              │
                                   ▼
                         アナリスト エージェント（LLM）
                         … BIが取った事実ベースの情報を入力として受け、
                           解釈・シナリオ・業種/個別の論点整理・投資テーゼの整理を行う
```

- **BI**: 「取得・正規化・再現可能な保存・他エージェントが引ける形」まで担当。
- **アナリスト**（主に業界・個別）: 生APIレスポンスを直接さばくのではなく、**BI経由で揃えた情報** を主入力にし、**LLMで分析・要約・仮説の整理** をする、という分工。
- **トレーダー**は、必要に応じてアナリストの出力・BIの定量と照らしてタイミング等を扱う（本メモの主役はアナリスト側）。

## 未決・あとで詰めること（メモ）

- ニュース・開示の **銘柄紐づけ・時系列** のキー設計（コード、日付、開示ID 等）。
- **データ鮮度・保持期間**（`CLAUDE.md` のアーカイブルールとの整合）。
- アナリストが読む **入力形式**（例: 要約済みチャンク、メタデータ付き JSON、RAG 用ストア など）は実装フェーズで選択。
- ニュースの **マクロ寄り** 論点は `macro_analyst` も同じ BI 出力を参照しうる（役割の被りをどう切るか）。
